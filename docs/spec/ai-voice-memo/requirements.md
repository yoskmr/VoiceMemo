## 1. 概要

### 1.1 プロダクトビジョン 🔵

**アプリ名**: Soyoka（つぶやき）

音声で手軽にメモ・日記を記録し、AIが自動で整理・要約・分析することで「録音→活用」のギャップを埋める iOS アプリ。日本語に特化したオンデバイス音声認識とプライバシーファーストの設計を核とする。

### 1.2 ターゲットユーザー 🔵

- **初期**: 開発者自身（個人開発・ドッグフーディング）
- **将来**: 思考整理・日記習慣を持つ一般ユーザーへ公開

### 1.3 プラットフォーム 🔵

- iOS のみ（MVP）
- SwiftUI による実装

### 1.4 ビジネスモデル 🔵

フリーミアム方式:

| プラン | 価格 | 内容 |
|:-------|:-----|:-----|
| 無料 | ¥0 | オンデバイスSTT無制限、録音・保存無制限、AI処理（要約・タグ付け・感情分析）合計月10回、基本検索、高度検索・フィルタ（タグ・感情・日付）、テーマカスタマイズ、アプリロック（Face ID/Touch ID）、ローカル保存 |
| Pro | ¥500/月 or ¥4,800/年 | AI処理無制限、きおくに聞く（AI対話）、こころの流れ（感情タイムライン+AIインサイト）、高精度仕上げ（クラウドLLMテキスト清書）、一週間をふりかえる（週次レポート）、きおくのつながり（関連メモ自動リンク）、Markdownエクスポート |

> ※v1.4対応: Pro機能再構成。無料プラン月10回→月10回に変更。テーマ・アプリロック・高度検索を無料化。クラウドSTTを高精度仕上げに再定義。新規Pro機能（きおくに聞く、こころの流れ、きおくのつながり）を追加。

## 3. 機能要件（EARS記法）

### EARS記法の分類

| 種別 | パターン | 意味 |
|:-----|:---------|:-----|
| Ubiquitous（通常） | The system shall ... | 常に満たすべき要件 |
| Event-driven（イベント駆動） | When [event], the system shall ... | 特定イベント発生時の要件 |
| State-driven（状態駆動） | While [state], the system shall ... | 特定の状態にある間の要件 |
| Optional（オプション） | Where [feature is included], the system shall ... | オプション機能の要件 |
| Unwanted behavior（制約） | If [unwanted condition], the system shall ... | 異常系・制約の要件 |

### 3.1 音声録音・文字起こし（P1）

#### REQ-001: ワンタップ録音開始 🔵

- **種別**: Ubiquitous
- **要件**: システムは、ホーム画面のワンタップ操作で音声録音を開始できなければならない。
- **根拠**: 音声メモの最も基本的な機能。録音開始までの摩擦を最小化する。
- **優先度**: P1

#### REQ-002: リアルタイム文字起こし表示 🔵

- **種別**: Event-driven
- **要件**: When 録音が開始されたとき, システムはリアルタイムでオンデバイス文字起こし結果を画面に表示しなければならない。
- **根拠**: ユーザーが認識精度を即座に確認できるようにする。
- **優先度**: P1

#### REQ-003: 録音完了後の自動AI要約 🔵

- **種別**: Event-driven
- **要件**: When 録音が完了したとき, システムは自動的にAI要約を生成しなければならない（無料プランは月10回上限）。ただし、無料プランかつ月次AI処理上限（REQ-011）に達している場合はREQ-011が優先され、AI要約の生成をスキップする。
- **根拠**: 「録音→活用」ギャップを埋めるコア機能。
- **優先度**: P3（AI要約フェーズ）
- ※Codexレビュー指摘 High#1 対応: REQ-011との優先規則を明記
- **AI処理カウント方法**: 1回の録音に対するAI処理（要約+タグ+感情分析）は統合プロンプトで同時実行され、月10回中 **1回** としてカウントする（REQ-004, REQ-005 と合わせて1回のAPI呼び出しで処理）。
- ※Phase 3 UXレビュー対応: AI処理のカウント方法を明記。統合プロンプトによる同時実行であることを明確化
- **カウント対象**: クラウドLLM（Backend Proxy経由）を利用した処理のみ月10回制限の対象。オンデバイスLLM（Apple Intelligence / llama.cpp）による処理はカウント対象外（無制限）。※Phase 3b product-owner 判断（2026-03-28）

#### REQ-004: タグ自動付与 🔵

- **種別**: Optional
- **要件**: Where Proプランが有効、または無料プランの月次AI処理枠（REQ-011: 月10回）が残存している場合, When AI要約が生成されたとき, システムはメモに対して内容に基づくタグを自動付与しなければならない。
- **根拠**: メモの整理・検索性を向上させる。
- **優先度**: P3
- ※Codexレビュー指摘 Critical#2 対応: Free/Pro機能分離を明確化。月10回制限がタグ付けにも適用されることを明記
- **AI処理カウント方法**: 1回の録音に対するAI処理（要約+タグ+感情分析）は統合プロンプトで同時実行され、月10回中 **1回** としてカウントする。タグ付けは要約（REQ-003）と同一のAPI呼び出し内で処理されるため、個別にはカウントしない。
- ※Phase 3 UXレビュー対応: AI処理のカウント方法を明記

#### REQ-005: 感情分析と可視化 🔵

- **種別**: Optional
- **要件**: Where Proプランが有効、または無料プランの月次AI処理枠（REQ-011: 月10回）が残存している場合, かつユーザーが感情分析を有効化しており（デフォルト: 無効）, When 文字起こしテキストが確定したとき, システムはテキストベースの感情分析を実施し、結果を可視化しなければならない。
- **根拠**: 日記・振り返り用途での自己理解を支援する。感情分析はクラウドLLM必須でAI処理回数を消費するため、ユーザーの明示的な同意を得るオプトイン方式とする。
- **優先度**: P3
- ※Codexレビュー指摘 Critical#2 対応: Free/Pro機能分離を明確化。月10回制限が感情分析にも適用されることを明記
- **AI処理カウント方法**: 1回の録音に対するAI処理（要約+タグ+感情分析）は統合プロンプトで同時実行され、月10回中 **1回** としてカウントする。感情分析はクラウドLLM必須であり、有効化時は統合プロンプトに含まれ月10回の1カウントに含まれる。無効化時は要約+タグのみで1カウントとなる。
- ※Phase 3 UXレビュー対応: 感情分析のオプトインとカウント方法の関係を明記

#### REQ-006: フルテキスト検索 🔵

- **種別**: Ubiquitous
- **要件**: システムは、保存されたすべてのメモに対してフルテキスト検索を提供しなければならない。
- **根拠**: 蓄積されたメモの活用にはすばやい検索が不可欠。
- **優先度**: P2

#### REQ-007: Markdownエクスポート 🔵

- **種別**: Optional
- **要件**: Where Proプランが有効な場合, システムはメモをMarkdown形式でエクスポートできなければならない。
- **根拠**: 外部ツール（Obsidian、Notion等）との連携を可能にする。
- **優先度**: P4

#### REQ-008: ローカル主保存 + クラウド処理時のデータ非保持 🔵

- **種別**: Ubiquitous
- **要件**: システムは、すべてのユーザーデータ（録音、文字起こし、メモ、設定）の永続保存先を端末内ローカルとしなければならない。クラウドLLM処理（REQ-010/012/018）においてテキストデータを送信する場合、以下のデータ取り扱いポリシーを遵守しなければならない: (1) 送信テキストのサーバー側非保持、(2) LLM学習データへの利用禁止、(3) 処理完了後の即時削除。なお、iOSのシステムバックアップ（iCloud）に関するポリシーはNFR-021で定義する。
- **根拠**: プライバシーファーストの設計思想。ユーザーの最大懸念であるプライバシーに対応。クラウド処理との論理的整合性を確保する。
- **優先度**: P1
- ※Codexレビュー指摘 Critical#1 対応
- ※Codex再レビュー v1.2対応: iCloudバックアップポリシーとの境界をNFR-021への参照として明記

#### REQ-009: オフラインSTT動作 🔵

- **種別**: State-driven
- **要件**: While ネットワーク接続がない状態でも, システムはオンデバイスSTTによる音声文字起こしを正常に実行できなければならない。
- **根拠**: 完全オンデバイスSTTの方針。通勤中や機内でも利用可能にする。
- **優先度**: P1

#### REQ-010: Backend Proxy経由のクラウドLLM処理 🔵

- **種別**: Ubiquitous（制約）
- **要件**: システムは、LLMのクラウド処理においてAPIキーを端末に保持せず、Backend Proxy（Cloudflare Workers等）経由で通信しなければならない。
- **根拠**: APIキー漏洩防止。最初からProxy構築する方針。
- **優先度**: P3
- ※Codexレビュー指摘 High#2 対応: Unwanted behaviorではなく設計制約としてUbiquitousに変更

#### REQ-011: 無料プランAI処理制限 🔵

- **種別**: Unwanted behavior（制約）
- **要件**: If 無料プランユーザーがAI処理（AI要約: REQ-003、タグ自動付与: REQ-004、感情分析: REQ-005）を月10回使用済みの場合, システムは追加のAI処理リクエストを拒否し、Proプランへのアップグレード案内を表示しなければならない。
- **根拠**: フリーミアムモデルの課金ゲート設計。毎日1回利用で約10日で枯渇し、残り20日間のPro転換欲求を最大化する設計。月10回では月末まで持ちこたえてしまい課金動機が弱まるため、10回に削減。
- **優先度**: P3
- ※Codexレビュー指摘 Critical#2 対応: 月10回制限がREQ-003/004/005すべてに適用されることを明記
- ※v1.4対応: 月10回→月10回に変更。課金転換率最大化のための調整

#### REQ-012: Proプラン無制限クラウドAI 🔵

- **種別**: Optional
- **要件**: Where Proプランが有効な場合, システムはクラウドAI機能（高精度STT、AI要約、タグ付け、感情分析）を回数制限なく提供しなければならない。
- **根拠**: Pro月額の価値提供。
- **優先度**: P3

#### REQ-013: 録音一時停止・再開 🟡

- **種別**: Event-driven
- **要件**: When ユーザーが録音中に一時停止ボタンをタップしたとき, システムは録音を一時停止し、再開ボタンで録音を継続できなければならない。
- **根拠**: 長時間の思考整理では中断が必要になる場面がある。
- **優先度**: P1

#### REQ-014: 録音の自動保存 🟡

- **種別**: Event-driven
- **要件**: When アプリがバックグラウンドに移行したとき, システムは録音中のデータを自動的に保存しなければならない。
- **根拠**: 録音データの消失はユーザー体験を大きく損なう。
- **優先度**: P1
- ※Codexレビュー指摘 High#2 対応: Unwanted behaviorではなく通常イベントとしてEvent-drivenに変更

#### REQ-015: メモ一覧表示 🔵

- **種別**: Ubiquitous
- **要件**: システムは、保存されたメモを時系列順の一覧で表示しなければならない。
- **根拠**: P2のメモ管理機能として明示されている。
- **優先度**: P2

#### REQ-016: メモ編集 🟡

- **種別**: Event-driven
- **要件**: When ユーザーがメモを選択したとき, システムは文字起こしテキスト・タイトル・タグを編集できるインターフェースを提供しなければならない。
- **根拠**: STTの誤認識修正やメモの補足が必要。
- **優先度**: P2

#### REQ-017: メモ削除 🟡

- **種別**: Event-driven
- **要件**: When ユーザーがメモの削除を要求したとき, システムは確認ダイアログを表示した後、メモとその関連データ（音声ファイル含む）を完全に削除しなければならない。
- **根拠**: データ管理の基本機能。プライバシー配慮から完全削除が必要。
- **優先度**: P2

#### REQ-018: 高精度仕上げ（Proプラン） 🔵

- **種別**: Optional
- **要件**: Where Proプランが有効な場合, システムはオンデバイスSTTの文字起こし結果をクラウドLLM経由で後処理し、固有名詞補正・句読点最適化・フィラー完全除去・文体整形を行う「高精度仕上げ」機能を提供しなければならない。
- **根拠**: 従来の「クラウド高精度STT」（音声ファイルをクラウドに送信）はプライバシーリスクが高い。代わりにテキストのみをクラウドLLMに送信する「仕上げ」方式に再定義。オンデバイスSTT + クラウドLLM後処理のハイブリッドで差別化する。
- **優先度**: P3
- **プライバシー**: 音声ファイルはクラウドに送信しない。STT結果テキストのみをBackend Proxy経由でLLM処理する。
- ※v1.4対応: 「クラウド高精度STT」→「高精度仕上げ」に再定義。音声送信型からテキスト送信型に変更。

#### REQ-019: 高度な検索・フィルタリング 🔵

- **種別**: Ubiquitous
- **要件**: システムはタグ・感情・日付範囲によるフィルタリングおよび高度な検索機能を提供しなければならない。
- **根拠**: 検索・フィルタリングはメモ活用のコア体験であり、有料ゲートにするとデータ活用体験が断絶する。無料ユーザーにも開放する。
- **優先度**: P3
- ※v1.4対応: Pro限定→無料に降格。検索はコア体験の一部として全ユーザーに提供。

#### REQ-020: テーマカスタマイズ 🔵

- **種別**: Optional
- **要件**: Where ユーザーが設定で選択した場合, システムはUIテーマのカスタマイズ機能（system / light / dark / journal）を提供しなければならない。
- **根拠**: 2026年にテーマカスタマイズが有料は時代にそぐわない。Day One、Bearなど競合も無料で提供しており、無料開放する。
- **優先度**: P4
- ※v1.4対応: Pro限定→無料に降格。

#### REQ-021: オンデバイスLLM基本処理 🔵

- **種別**: Optional
- **要件**: Where デバイスがA16 Bionic以降かつメモリ6GB以上の場合, システムは基本的なLLM処理（簡易要約・キーワード抽出等）をオンデバイスで実行できなければならない。Where デバイスが上記要件を満たさない場合, システムはクラウドLLM（REQ-010経由）にフォールバックしなければならない。
- **根拠**: LLMハイブリッド方針のうちオンデバイス部分。デバイス性能によりオンデバイス処理の可否が分かれるため、Optional種別とする。
- **優先度**: P3
- ※Codexレビュー指摘 High#4 対応: 対象デバイス要件およびフォールバック方針を追記
- ※Codex再レビュー v1.2対応: 種別をUbiquitousからOptionalに変更。非対応端末フォールバックがあるため、EARS記法のWhereパターンに適合するOptionalが妥当

#### REQ-022: 感情分析結果の時系列可視化 🟡

- **種別**: Ubiquitous
- **要件**: システムは、感情分析結果を時系列グラフとして表示し、ユーザーの感情推移を可視化しなければならない。
- **根拠**: 日記・振り返り用途において感情の変化を把握する価値がある。
- **優先度**: P3

#### REQ-023: 音声再生 🟡

- **種別**: Event-driven
- **要件**: When ユーザーがメモ詳細画面で再生ボタンをタップしたとき, システムは録音された音声を再生できなければならない。
- **根拠**: 文字起こしの確認や聞き返しのために元音声の再生は必要。
- **優先度**: P2

#### REQ-024: アプリ内課金（In-App Purchase） 🔵

- **種別**: Ubiquitous
- **要件**: システムは、Apple In-App Purchase を通じてProプラン（月額¥500 / 年額¥4,800）のサブスクリプションを提供しなければならない。
- **根拠**: フリーミアムビジネスモデルの実装。
- **優先度**: P3

#### REQ-025: 日本語STT最適化 🟡

- **種別**: Ubiquitous
- **要件**: システムは、日本語の音声認識において固有名詞・専門用語のカスタム辞書登録機能を提供しなければならない。
- **根拠**: 市場調査で「日本語STT精度」がユーザー最大のペインとして特定されている。
- **優先度**: P2

#### REQ-026: 録音時間表示 🟡

- **種別**: State-driven
- **要件**: While 録音中の間, システムは経過時間をリアルタイムで表示しなければならない。
- **根拠**: ユーザーが録音の長さを把握するための基本UI。
- **優先度**: P1

#### REQ-027: アプリ起動時の認証 🔴

- **種別**: Optional
- **要件**: Where ユーザーが設定で有効化した場合, システムはアプリ起動時にFace ID/Touch IDによる認証を要求しなければならない。
- **根拠**: 日記アプリはプライベートな内容を含むため、端末ロックとは別のアプリレベル認証が望ましい。プライバシーファーストを標榜するアプリとして、プライバシー保護機能を有料にするのは矛盾するため無料開放する。
- **優先度**: P4
- ※v1.4対応: Pro限定→無料に降格。プライバシーファーストの一貫性を確保。

#### REQ-028: 複数エクスポート先対応 🔴

- **種別**: Optional
- **要件**: Where Proプランが有効な場合, システムはMarkdownエクスポートに加え、共有シート経由で他アプリへの連携を可能にしなければならない。
- **根拠**: iOS標準の共有機能との統合は自然な拡張。
- **優先度**: P4

#### REQ-029: ウィジェット対応 🔴

- **種別**: Optional
- **要件**: Where iOSウィジェットが有効な場合, システムはホーム画面ウィジェットからワンタップで録音を開始できなければならない。
- **根拠**: 「ワンタップ録音」をさらに強化する手段。ホーム画面からの即座のアクセスを実現。
- **優先度**: P4

#### REQ-030: 利用統計ダッシュボード 🔴

- **種別**: Ubiquitous
- **要件**: システムは、録音回数・総録音時間・メモ数・感情推移等の利用統計をダッシュボードで表示しなければならない。
- **根拠**: 日記・振り返り用途において習慣化を促進するフィードバック機能。
- **優先度**: P4

#### REQ-031: きおくに聞く（AI対話） 🟡

- **種別**: Optional
- **要件**: Where Proプランが有効な場合, システムは蓄積されたメモに対してユーザーが自然言語で質問し、AIが過去のメモを参照して回答する対話機能を提供しなければならない。
- **根拠**: 「録音→活用」のギャップを埋めるプロダクトビジョンの直接的実現。蓄積された「きおく」に対して「先週何に悩んでいた？」「最近よく考えていることは？」と質問できる体験は競合にない独自価値。RAG（検索拡張生成）でFTS5既存検索基盤とクラウドLLMを組み合わせて実現する。
- **優先度**: P3
- **技術方針**: SwiftData + FTS5 の既存検索基盤で関連メモを検索し、Backend Proxy経由のクラウドLLMで回答を生成する。Streaming Response対応。

#### REQ-032: こころの流れ（感情タイムライン + AIインサイト） 🟡

- **種別**: Optional
- **要件**: Where Proプランが有効な場合, システムは感情分析結果の長期トレンドを可視化し、AIが生成するインサイト（例:「先月は不安が多かったけど、今週は穏やかな傾向です」）を提供しなければならない。
- **根拠**: REQ-022（感情分析結果の時系列可視化）の拡張。単なるグラフ表示ではなく、AIによる言語化されたインサイトを付加することで、感情の自己理解を深める独自価値を提供する。REQ-030（利用統計ダッシュボード）と統合し、「あなたの1ヶ月」「あなたの3ヶ月」のような振り返りビューを構築する。
- **優先度**: P3
- **関連REQ**: REQ-005, REQ-022, REQ-030

#### REQ-033: きおくのつながり（関連メモ自動リンク） 🟡

- **種別**: Optional
- **要件**: Where Proプランが有効な場合, システムはAI処理時に過去メモとのセマンティック類似度を計算し、関連するメモを自動リンクする機能を提供しなければならない。
- **根拠**: メモが蓄積されるほど「あのとき何を考えていたか」が埋もれる。Obsidianのバックリンクの音声メモ版として、思考の軌跡を可視化する。競合の音声メモアプリでセマンティック関連付けを実装しているものはない。
- **優先度**: P3-P4
- **技術方針**: CoreML の Sentence Embedding でオンデバイス類似度計算を検討。SwiftDataにベクトルインデックスを追加する方式。

## 4. 非機能要件

### 4.1 パフォーマンス

#### NFR-001: 録音開始レイテンシ 🟡

- **要件**: 録音開始ボタンタップから実際の録音開始まで 500ms 以内であること。
- **測定条件**: iPhone 15, iOS 17以上, ネットワーク非接続時。基準: p95。
- **根拠**: 音声メモの即時性を確保するため。
- ※Codexレビュー指摘 Medium#1 対応: 測定条件を追記

#### NFR-002: STTレイテンシ 🟡

- **要件**: オンデバイスSTTのリアルタイム文字起こし遅延は 2秒以内であること。
- **測定条件**: iPhone 15, iOS 17以上, ネットワーク非接続時（オンデバイスSTT）。基準: p95。
- **根拠**: リアルタイム表示として許容される範囲。
- ※Codexレビュー指摘 Medium#1 対応: 測定条件を追記

#### NFR-003: AI要約生成時間 🟡

- **要件**: 5分以内の録音に対するAI要約生成は 10秒以内に完了すること。
- **測定条件**: iPhone 15, iOS 17以上, Wi-Fi接続時（クラウドLLM）/ ネットワーク非接続時（オンデバイスLLM）。基準: p95。
- **根拠**: ユーザーが待機を許容できる時間。
- ※Codexレビュー指摘 Medium#1 対応: 測定条件を追記

#### NFR-004: アプリ起動時間 🟡

- **要件**: コールドスタートでのアプリ起動は 2秒以内であること。
- **測定条件**: iPhone 15, iOS 17以上, ネットワーク非接続時。基準: p95。
- **根拠**: iOS標準的なユーザー期待値。
- ※Codexレビュー指摘 Medium#1 対応: 測定条件を追記

#### NFR-005: メモ一覧ロード 🟡

- **要件**: 1,000件のメモ一覧表示は 1秒以内にロードすること。
- **測定条件**: iPhone 15, iOS 17以上, ネットワーク非接続時。基準: p95。
- **根拠**: 日常利用で蓄積されるメモ量を想定。
- ※Codexレビュー指摘 Medium#1 対応: 測定条件を追記

#### NFR-006: ストレージ効率 🟡

- **要件**: 1分の録音データは圧縮後 500KB 以内に収めること。
- **測定条件**: AAC/OPUS圧縮、サンプルレート44.1kHz、モノラル。基準: p95。
- **根拠**: 端末ストレージへの影響を最小限にする。
- ※Codexレビュー指摘 Medium#1 対応: 測定条件を追記

### 4.2 セキュリティ

#### NFR-007: データ暗号化 🟡

- **要件**: 端末内に保存されるすべてのデータは、iOS Data Protection（NSFileProtectionCompleteUntilFirstUserAuthentication）により暗号化されること。これにより、端末起動後（初回ロック解除後）はバックグラウンド録音中もファイルアクセスが可能となる。NSFileProtectionCompleteは端末ロック時にファイルアクセス不可となりバックグラウンド録音（EC-003）と衝突するため採用しない。
- **根拠**: ローカル主保存方針のセキュリティ補強。バックグラウンド録音との両立を考慮。
- ※Codexレビュー指摘 High#3 対応: NSFileProtectionCompleteからNSFileProtectionCompleteUntilFirstUserAuthenticationに変更

#### NFR-008: APIキー非保持 🔵

- **要件**: アプリバイナリおよび端末ストレージにクラウドサービスのAPIキーを保持しないこと。
- **根拠**: Backend Proxy方針として明示されている。

#### NFR-009: 通信暗号化 🟡

- **要件**: Backend Proxyとの通信はすべてTLS 1.3以上で暗号化すること。
- **根拠**: クラウド通信時のセキュリティ標準。

#### NFR-010: 認証トークン管理 🟡

- **要件**: Backend Proxyへのアクセスにはユーザー認証トークン（Keychain保存）を使用すること。
- **根拠**: 不正利用防止。

#### NFR-011: プライバシーポリシー準拠 🔴

- **要件**: Apple App Store Review Guidelinesおよび日本の個人情報保護法に準拠したプライバシーポリシーを提供すること。
- **根拠**: 一般公開時の法的要件。

### 4.3 ユーザビリティ

#### NFR-012: 感性的ジャーナルUI 🔵

- **要件**: UIデザインは温かみのあるジャーナル風デザインとすること。具体的な評価指標: カラーパレットはHSB色相20-40の暖色系を基調、角丸半径12pt以上、フォントは丸ゴシック系を使用する。
- **根拠**: UIデザイン方針として明示されている。
- ※Codexレビュー指摘 Low#2 対応: 主観的表現を検証可能な評価指標に変更

#### NFR-013: ダークモード対応 🟡

- **要件**: iOS標準のダークモードに対応すること。
- **根拠**: iOS開発の標準的なアクセシビリティ対応。

#### NFR-014: 操作ステップ最小化 🟡

- **要件**: 「録音開始→保存」までの操作は最大3タップ以内で完了できること。
- **根拠**: 音声メモの本質的な価値である手軽さの確保。

#### NFR-015: アクセシビリティ 🔴

- **要件**: VoiceOver、Dynamic Typeに対応すること。
- **根拠**: iOS標準のアクセシビリティガイドライン準拠。

#### NFR-016: オンボーディング 🔴

- **要件**: 初回起動時に3画面以内のオンボーディングフローを表示し、マイク権限の取得を案内すること。
- **根拠**: STTアプリではマイク権限が必須であり、初回UXが重要。

### 4.4 信頼性

#### NFR-017: 録音データ保全 🟡

- **要件**: アプリクラッシュ時でも録音中のデータが失われないよう、定期的（5秒間隔）に中間保存すること。
- **根拠**: 録音データの消失はユーザー体験を致命的に損なう。

#### NFR-018: オフライン完全動作 🔵

- **要件**: ネットワーク接続がない状態でも、録音・STT・保存・閲覧・基本検索のすべてが動作すること。
- **根拠**: 完全オンデバイスSTT方針およびローカル保存方針。

### 4.5 保守性

#### NFR-019: STTエンジン差し替え可能性 🟡

- **要件**: STTエンジン（Apple SpeechAnalyzer / Whisper.cpp）をプロトコル抽象化し、将来的なエンジン追加・差し替えを容易にすること。
- **根拠**: 2026年春のApple Siri刷新リスクへの対応。技術変化に柔軟に対応する設計。

#### NFR-020: LLMプロバイダ差し替え可能性 🟡

- **要件**: LLMプロバイダ（オンデバイス/クラウド各種）をプロトコル抽象化し、プロバイダの追加・差し替えを容易にすること。
- **根拠**: LLM市場の急速な変化に対応する設計。

#### NFR-021: iCloudバックアップからの除外 🔴

- **要件**: 音声データ・文字起こしデータ・LLM処理結果はiCloudバックアップ対象外（`isExcludedFromBackup = true`）とする。ただし、アプリ設定（テーマ・言語設定等のメタデータ）はiCloudバックアップの対象とし、ユーザーの端末移行をサポートする。このメタデータにはユーザーの録音内容・個人情報を含まない。
- **根拠**: プライバシーファーストの設計思想。音声・文字起こしデータおよびLLM処理結果がiCloud経由で意図せずクラウドに保存されることを防ぐ。一方、端末移行時の利便性を確保するためアプリ設定メタデータはバックアップ対象とする。
- ※Codexレビュー指摘 Low#1 対応
- ※Codex再レビュー v1.2対応: LLM処理結果をバックアップ除外対象に追加。メタデータの範囲を厳密化し、個人情報を含まないことを明記。REQ-008との境界を明確化

## 5. エッジケース

### EC-001: 長時間録音 🟡

- **条件**: ユーザーが60分以上の連続録音を行った場合
- **対応**: システムは録音を継続しつつ、メモリ使用量を一定以下に保つためストリーミング方式で音声ファイルを書き出す。文字起こしは5分単位でチャンク処理する。
- **信頼性**: 🟡

### EC-002: ストレージ不足 🟡

- **条件**: 端末のストレージ残量が100MB未満の場合
- **対応**: システムは録音開始前に警告を表示し、古いメモの音声データ削除を提案する。
- **信頼性**: 🟡

### EC-003: バックグラウンド遷移中の録音 🟡

- **条件**: 録音中にアプリがバックグラウンドに移行した場合
- **対応**: システムはバックグラウンドオーディオセッションを維持し、録音を継続する。10分以上バックグラウンドの場合は自動保存して録音を終了する。
- **信頼性**: 🟡

### EC-004: STT認識不能音声 🟡

- **条件**: 環境ノイズが大きく、STTが有意なテキストを生成できない場合
- **対応**: システムは「音声を認識できませんでした」と表示し、録音データ自体は保存する。
- **信頼性**: 🟡

### EC-005: ネットワーク切断中のクラウドAI要求 🟡

- **条件**: ProプランユーザーがクラウドAI機能を要求したがネットワーク接続がない場合
- **対応**: システムはオンデバイスLLMにフォールバックし、「オンデバイスで処理しました。ネットワーク接続後にクラウド処理で再生成できます」と通知する。
- **信頼性**: 🟡

### EC-006: サブスクリプション有効期限切れ 🟡

- **条件**: Proプランの有効期限が切れた場合
- **対応**: システムはPro機能を制限するが、既に生成済みのAI要約・タグ・感情分析データはそのまま閲覧可能とする。エクスポート済みデータも維持する。
- **信頼性**: 🟡

### EC-007: 多言語混在音声 🔴

- **条件**: 日本語と英語が混在した音声入力の場合
- **対応**: システムは主言語を日本語として処理し、英語の固有名詞や単語が含まれる場合は可能な範囲でそのまま認識する。
- **信頼性**: 🔴

### EC-008: 同時操作 🔴

- **条件**: 録音中にメモ一覧の閲覧・検索を行った場合
- **対応**: システムは録音を継続しながら、他の画面への遷移・操作を許可する。録音状態はステータスバーで常時表示する。
- **信頼性**: 🔴

### EC-009: 大量メモ蓄積 🟡

- **条件**: メモが10,000件以上蓄積された場合
- **対応**: システムは遅延ロード（ページネーション）を適用し、一覧表示のパフォーマンスを維持する。検索インデックスはバックグラウンドで更新する。
- **信頼性**: 🟡

### EC-010: Backend Proxy障害 🟡

- **条件**: Backend Proxyがダウンまたは応答遅延（10秒以上）の場合
- **対応**: システムはタイムアウト後にオンデバイスLLMへフォールバックし、リトライ可能な旨をユーザーに通知する。
- **信頼性**: 🟡

### EC-011: マイク権限拒否時の動作 🟡

- **条件**: ユーザーがマイク権限を拒否または取り消した場合
- **対応**: システムは録音機能を無効化し、設定アプリへの誘導リンク付きで権限が必要な旨を表示する。既存メモの閲覧・検索は引き続き利用可能とする。
- **信頼性**: 🟡
- ※Codexレビュー指摘 Medium#2 対応

### EC-012: 音声認識権限拒否時の動作 🟡

- **条件**: ユーザーが音声認識権限を拒否または取り消した場合
- **対応**: システムは録音自体は許可するが、リアルタイム文字起こしを無効化する。録音完了後に「音声認識権限を許可すると文字起こしが利用できます」と案内を表示する。
- **信頼性**: 🟡
- ※Codexレビュー指摘 Medium#2 対応

### EC-013: 通話着信による録音中断 🟡

- **条件**: 録音中に電話の着信があった場合
- **対応**: システムはオーディオセッションの中断通知を受け取り、録音中のデータを即座に中間保存する。通話終了後、「録音が中断されました。続きを録音しますか？」と通知し、新規録音として再開できるようにする。
- **信頼性**: 🟡
- ※Codexレビュー指摘 Medium#2 対応

### EC-014: 月次リセット時刻の定義 🟡

- **条件**: 無料プランの月次AI処理回数（REQ-011）がリセットされるタイミング
- **対応**: システムは毎月1日 JST 0:00（UTC+9 00:00:00）にAI処理回数カウンターをリセットする。リセット処理はアプリ起動時にローカルで判定する。
- **信頼性**: 🟡
- ※Codexレビュー指摘 Medium#2 対応

### EC-015: サブスクリプション検証失敗時の猶予期間 🟡

- **条件**: App Storeへのサブスクリプション検証がネットワーク障害等で失敗した場合
- **対応**: システムは最後に検証成功した日時から72時間の猶予期間を設け、その間はPro機能を維持する。猶予期間超過後はPro機能を制限し、「サブスクリプションの確認にはインターネット接続が必要です」と表示する。
- **信頼性**: 🟡
- ※Codexレビュー指摘 Medium#2 対応

## 7. 用語集

| 用語 | 定義 |
|:-----|:-----|

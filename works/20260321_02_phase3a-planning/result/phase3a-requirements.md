## 1. Phase 3a スコープ定義

### 1.1 対象範囲

| カテゴリ | 対象 |
|:---------|:-----|
| **LLM エンジン** | Apple Intelligence Foundation Models API（iOS 26+）、llama.cpp Phi-3-mini Q4_K_M（フォールバック） |
| **AI 処理内容** | 要約生成（3行以内）、キーポイント抽出（最大5件）、タグ提案（最大3件） |
| **月次制限** | 無料プラン月15回（ローカルカウント管理） |
| **処理基盤** | AI 処理キュー（バックグラウンド対応） |
| **永続化** | AI 処理結果の SwiftData 保存 |
| **UI** | メモ詳細画面の AI 要約・タグ表示の実体化 |

### 1.2 スコープ外（Phase 3b 以降）

| カテゴリ | 理由 |
|:---------|:-----|
| 感情分析（REQ-005） | クラウド LLM 必須。オンデバイスでは対応不可 |
| Backend Proxy（REQ-010） | Phase 3b で構築 |
| クラウド LLM | Backend Proxy 経由が必須（REQ-010） |
| クラウド高精度 STT（REQ-018） | Pro プラン + Backend 必要 |
| StoreKit 課金（REQ-024） | Phase 3c で実装 |
| 感情時系列可視化（REQ-022） | 感情分析データが前提 |
| 高度検索・フィルタリング（REQ-019） | Pro プラン機能 |

## 2. 機能要件（EARS 記法）

### P3A-REQ-001: 録音完了後のオンデバイス AI 要約生成

- **種別**: Event-driven
- **要件**: When 録音が完了し、文字起こしテキストが確定したとき, システムはオンデバイス LLM を使用して AI 要約（タイトル 20 文字以内、要約文 1-2 行、キーポイント最大 5 件）を自動生成しなければならない。ただし、無料プランかつ月次 AI 処理上限（P3A-REQ-007）に達している場合は AI 要約生成をスキップする。
- **根拠**: REQ-003（録音完了後の自動 AI 要約）のオンデバイス実装。Backend 不要で AI の価値をユーザーに提供する。
- **親要件**: REQ-003, REQ-021
- **入力条件**: 文字起こしテキストが 10 文字以上であること。10 文字未満の場合は「テキストが短すぎるため要約を生成できません」と表示する。
- **出力形式**: JSON（`{"title": "...", "brief": "...", "keyPoints": [...]}` ）

### P3A-REQ-002: オンデバイス LLM によるタグ自動生成

- **種別**: Event-driven
- **要件**: When AI 要約が生成されたとき（P3A-REQ-001 と同一のプロンプト実行内）, システムはメモの内容に基づくタグを最大 3 件自動生成し、メモに付与しなければならない。
- **根拠**: REQ-004（タグ自動付与）のオンデバイス実装。要約とタグは統合プロンプトで同時生成するため、追加の LLM 呼び出しは不要。
- **親要件**: REQ-004, REQ-021
- **出力形式**: JSON（`{"tags": ["タグ1", "タグ2", "タグ3"]}` ）。タグラベルは 15 文字以内。
- **AI 処理カウント**: 要約生成（P3A-REQ-001）と同一の処理内で実行されるため、個別にはカウントしない。

### P3A-REQ-003: Apple Intelligence Foundation Models API の優先利用

- **種別**: State-driven
- **要件**: While デバイスが iOS 26 以降かつ Apple Intelligence が利用可能な状態にあるとき, システムは Apple Intelligence Foundation Models API を LLM プロバイダとして優先利用しなければならない。
- **根拠**: Apple 純正エンジンは Neural Engine 最適化により低消費電力・高速で動作する。ユーザーのバッテリー・発熱への配慮。
- **親要件**: REQ-021
- **制約**: Apple Intelligence Foundation Models API は `@Generable` マクロ経由の構造化出力のみ対応。自由プロンプト実行が不可の場合は P3A-REQ-004（llama.cpp フォールバック）を適用する。

### P3A-REQ-004: llama.cpp フォールバック

- **種別**: State-driven
- **要件**: While Apple Intelligence Foundation Models API が利用不可（iOS 25 以前、または API 制約により構造化出力のみ対応）の状態であり、かつデバイスが A16 Bionic 以降かつメモリ 6GB 以上であるとき, システムは llama.cpp (Phi-3-mini Q4_K_M) をオンデバイス LLM プロバイダとして使用しなければならない。
- **根拠**: Apple Intelligence 非対応端末でもオンデバイス AI 体験を提供する。llama.cpp は自由プロンプト実行が可能で、要約+タグ統合プロンプトに対応。
- **親要件**: REQ-021
- **メモリ要件**: モデルファイル約 2.5GB。STT モデルをアンロード後に LLM を起動する排他制御が必要。
- **モデル配置**: `Library/Caches/Models/phi-3-mini-q4_k_m.gguf`（再ダウンロード可能領域）

### P3A-REQ-005: オンデバイス LLM 非対応端末のグレースフル・デグラデーション

- **種別**: Unwanted behavior
- **要件**: If デバイスが A16 Bionic 未満またはメモリ 6GB 未満であり、かつ Apple Intelligence も利用不可の場合, システムは AI 処理をスキップし、「このデバイスでは AI 分析をご利用いただけません。対応デバイスについてはこちら」というメッセージを表示しなければならない。AI 処理以外の機能（録音、文字起こし、メモ管理、検索）は通常通り利用可能とする。
- **根拠**: REQ-021（非対応端末はクラウド LLM にフォールバック）の Phase 3a 限定対応。Phase 3a では Backend が存在しないため、クラウドフォールバックは不可。Phase 3b で Backend 構築後にクラウドフォールバックを有効化する。
- **親要件**: REQ-021

### P3A-REQ-006: AI 処理キューの実装

- **種別**: Ubiquitous
- **要件**: システムは、AI 処理リクエストをキューで管理し、以下の要件を満たさなければならない:
  1. 録音完了時に AI 処理タスクを自動的にキューに追加する
  2. キュー内のタスクは優先度順（高 > 通常 > 低）で処理する
  3. 処理中のタスクのステータス（queued / processing / completed / failed）をリアルタイムで UI に反映する
  4. 処理失敗時は最大 3 回のリトライ（指数バックオフ: 2秒、4秒、8秒）を行う
  5. アプリがバックグラウンドに移行しても、実行中の AI 処理は継続する
- **根拠**: AI 処理はオンデバイス LLM で数秒かかるため、バックグラウンドでの非同期処理が必要。ユーザーは AI 処理完了を待たずに他の操作を行える。
- **親要件**: 設計書 DES-002 セクション5（AI 処理キュー設計）

### P3A-REQ-007: 月次 AI 処理回数のローカルカウント管理

- **種別**: Unwanted behavior
- **要件**: If 無料プランユーザーが月15回の AI 処理を使用済みの場合, システムは追加の AI 処理リクエストを拒否し、以下の情報を表示しなければならない:
  1. 「今月の AI 処理回数に到達しました」メッセージ
  2. 次回リセット日（翌月1日 JST 0:00）
  3. Pro プランへの案内リンク（Phase 3c で課金実装後に有効化。Phase 3a ではプレースホルダ）
- **根拠**: REQ-011（無料プラン AI 処理制限）のローカル実装。Phase 3a では Backend が存在しないため、SwiftData で月次カウントを管理する。
- **親要件**: REQ-011
- **カウント方法**: 1回の録音に対する AI 処理（要約+タグ）は統合プロンプトで同時実行され、月15回中 **1回** としてカウントする。
- **リセットタイミング**: 毎月1日 JST 0:00（UTC+9 00:00:00）。アプリ起動時にローカルで判定（EC-014 準拠）。

### P3A-REQ-008: AI 処理結果の SwiftData 永続化

- **種別**: Event-driven
- **要件**: When AI 処理が完了したとき, システムは以下のデータを SwiftData に永続化しなければならない:
  1. **要約データ**: タイトル、要約テキスト、キーポイント配列、生成日時、プロバイダ種別、オンデバイスフラグ
  2. **タグデータ**: タグ名、生成元（`ai`）、作成日時
  3. **処理メタデータ**: 処理時間（ms）、使用プロバイダ、処理ステータス
- **根拠**: AI 処理結果はメモの永続的な付加価値であり、再生成なしで参照できる必要がある。
- **親要件**: REQ-008（ローカル主保存）

### P3A-REQ-009: メモ詳細画面での AI 要約表示

- **種別**: Ubiquitous
- **要件**: システムは、メモ詳細画面で AI 要約を以下のレイアウトで表示しなければならない:
  1. AI 要約カード: 要約テキスト + キーポイント（展開/折りたたみ対応）
  2. 処理場所バッジ: 「オンデバイス処理」（Phase 3a では常にオンデバイス）
  3. タグチップ: AI 生成タグを横スクロール表示
  4. AI 要約が未生成の場合: プレースホルダ表示（「AI 要約はまだ生成されていません」）
- **根拠**: 既存の UI コンポーネント（`AISummarySection`, `TagFlowLayout`）を実データで動作させる。
- **親要件**: REQ-003, REQ-004, 設計書 DESIGN-004 セクション4.5, セクション14.3

### P3A-REQ-010: AI 処理ステータスのリアルタイム表示

- **種別**: State-driven
- **要件**: While AI 処理が実行中の状態にあるとき, システムはメモ詳細画面に以下の情報をリアルタイムで表示しなければならない:
  1. 処理進捗（0%-100% のプログレスバー）
  2. 処理段階の説明（「要約・タグを生成しています」）
  3. 今月の利用回数（「今月の利用: X/15回」）
- **根拠**: 設計書 DESIGN-004 セクション14.1 準拠。オンデバイス LLM は処理に数秒かかるため、進捗表示によりユーザーの待機不安を軽減する。
- **親要件**: 設計書 DESIGN-004 セクション14

### P3A-REQ-011: AI 処理の手動リトライ

- **種別**: Event-driven
- **要件**: When AI 処理が失敗し、ユーザーが「リトライ」ボタンをタップしたとき, システムは AI 処理を再実行しなければならない。手動リトライは月15回制限のカウントに含まれない（失敗した処理の再試行であるため）。
- **根拠**: オンデバイス LLM はメモリ不足等で失敗する可能性があり、ユーザーに再試行の手段を提供する必要がある。
- **親要件**: REQ-003

### P3A-REQ-012: AI 処理の手動トリガー（再生成）

- **種別**: Event-driven
- **要件**: When ユーザーがメモ詳細画面の AI 要約カードにある「再生成」ボタンをタップしたとき, システムは AI 要約・タグを再生成しなければならない。再生成は月15回制限の **1回としてカウント** する。
- **根拠**: 文字起こしテキストを手動編集した後に、更新されたテキストで AI 要約を再生成したいユースケースに対応する。
- **親要件**: REQ-003

### P3A-REQ-013: オンデバイス LLM モデルのダウンロード管理

- **種別**: Event-driven
- **要件**: When アプリが初回起動またはモデルファイルが存在しないとき, システムは以下の手順でオンデバイス LLM モデルをダウンロードしなければならない:
  1. ダウンロードの必要性をユーザーに通知する（「AI 分析を利用するにはモデルのダウンロードが必要です（約 2.5GB）」）
  2. Wi-Fi 接続を推奨する（「Wi-Fi 接続でのダウンロードを推奨します」）
  3. ダウンロード進捗を表示する
  4. ダウンロード完了後に「AI 分析が利用可能になりました」と通知する
  5. ダウンロード完了まで AI 処理はスキップし、メモの録音・保存は通常通り利用可能とする
- **根拠**: Phi-3-mini Q4_K_M は約 2.5GB あり、初回のダウンロードが必要。ユーザーが意図せず大量のデータ通信を発生させないよう、明示的な案内を行う。
- **適用条件**: llama.cpp フォールバック利用時のみ。Apple Intelligence 利用時はシステムが言語モデルを管理するため不要。

### P3A-REQ-014: LLM とSTT のメモリ排他制御

- **種別**: Ubiquitous
- **要件**: システムは、STT エンジンとオンデバイス LLM のメモリ使用を排他的に管理しなければならない。具体的には:
  1. STT 処理中にオンデバイス LLM を同時に起動しない
  2. 録音完了後、STT モデルをアンロードしてからオンデバイス LLM を起動する
  3. オンデバイス LLM 処理完了後、LLM モデルをアンロードする
- **根拠**: 設計書 DES-002 セクション7.4 準拠。iPhone 15（6GB RAM）で STT（~800MB）と LLM（~2.5GB）を同時に保持するとメモリ上限を超過する。
- **親要件**: 設計書 DES-002 セクション7.3, 7.4

### P3A-REQ-015: 月次利用残回数の事前警告

- **種別**: State-driven
- **要件**: While 無料プランの月次 AI 処理残回数が 3 回以下であるとき, システムはメモ一覧画面上部に残回数をプログレスバーで表示し、以下の段階的な警告を行わなければならない:
  1. 残り 3 回: プログレスバーが警告色（vmWarning）に変化
  2. 残り 1 回: 「今月の AI 処理はあと 1 回です」メッセージ表示
  3. 残り 0 回: プログレスバーがエラー色（vmError）に変化 + 「来月1日にリセットされます」テキスト
- **根拠**: 設計書 DESIGN-004 セクション14.2 準拠。突然の制限到達ではなく、段階的な通知によりユーザー体験を滑らかにする。
- **親要件**: REQ-011

### P3A-REQ-016: 初回 AI 処理時のオンボーディング

- **種別**: Event-driven
- **要件**: When ユーザーが初めて AI 処理が実行される場面に遭遇したとき（初回の録音完了時）, システムは以下の情報を含むオンボーディングシートを表示しなければならない:
  1. AI 分析機能の説明（「録音内容を AI が自動で要約・タグ付けします」）
  2. オンデバイス処理であることの説明（「この処理はお使いのデバイス上で行われます。テキストは外部に送信されません」）
  3. 月15回の無料枠がある説明（「毎月15回まで無料でご利用いただけます」）
  4. 「はじめる」ボタンで閉じる
- **根拠**: AI 処理に対するユーザーの理解とプライバシーへの安心感を初回体験で確立する。
- **親要件**: NFR-016（オンボーディング）

## 3. 非機能要件

### P3A-NFR-001: オンデバイス LLM レスポンス時間

- **要件**: 500 文字以内のテキストに対するオンデバイス LLM の要約+タグ生成は 5 秒以内に完了すること。
- **測定条件**: iPhone 15, iOS 17 以上, llama.cpp Phi-3-mini Q4_K_M。基準: p95。
- **根拠**: NFR-003 準拠。ユーザーが待機を許容できる時間。
- **対応 NFR**: NFR-003

### P3A-NFR-002: オンデバイス LLM メモリ使用量

- **要件**: オンデバイス LLM（llama.cpp Phi-3-mini Q4_K_M）の推論時メモリ使用量は 2.5GB 以下であること。アプリ全体のメモリ使用量（LLM 推論中）は 3.2GB 以下であること。
- **測定条件**: iPhone 15（6GB RAM）。STT モデルアンロード済み。
- **根拠**: 設計書 DES-002 セクション7.3 準拠。

### P3A-NFR-003: AI 処理キューの永続性

- **要件**: AI 処理キューのタスクは SwiftData に永続化され、アプリのクラッシュまたは強制終了後も処理キューが復元されること。
- **根拠**: AI 処理は録音の付加価値であり、キューの消失はユーザー体験を損なう。

### P3A-NFR-004: オンデバイス LLM のバッテリー消費

- **要件**: 5 分の録音に対する AI 処理のバッテリー消費は 1% 以下であること。
- **測定条件**: iPhone 15, バッテリー残量 50% 以上。
- **根拠**: ユーザーが AI 処理によるバッテリー消耗を意識しない水準を目指す。

### P3A-NFR-005: プロンプトの出力品質

- **要件**: オンデバイス LLM の出力が有効な JSON 形式である確率は 95% 以上であること。JSON パース失敗時はリトライ（最大 2 回）を行い、最終的に失敗した場合はエラーとしてユーザーに通知する。
- **測定条件**: 日本語テキスト 100-500 文字, llama.cpp Phi-3-mini Q4_K_M, temperature=0.3。
- **根拠**: LLM の出力は JSON 形式を期待するが、オンデバイスモデルは出力の安定性がクラウドモデルより低い。

## 4. エッジケース

### P3A-EC-001: 文字起こしテキストが空の場合

- **条件**: 録音完了したが、STT が有意なテキストを生成できなかった場合（EC-004 に該当）
- **対応**: AI 処理をスキップし、「音声を認識できなかったため、AI 分析をスキップしました」と表示する。月15回制限のカウントは消費しない。

### P3A-EC-002: オンデバイス LLM モデルのダウンロード失敗

- **条件**: モデルファイルのダウンロードがネットワーク切断等で失敗した場合
- **対応**: 「モデルのダウンロードに失敗しました。Wi-Fi に接続して再試行してください」と表示する。ダウンロード完了まで AI 処理はスキップする。録音・文字起こし・メモ管理は通常通り利用可能。

### P3A-EC-003: LLM 出力が不正な JSON の場合

- **条件**: オンデバイス LLM が有効な JSON を返さなかった場合
- **対応**: 最大 2 回のリトライ（同一プロンプトで再実行）を行う。全リトライ失敗後、「AI 分析に失敗しました。もう一度お試しください」と表示する。月15回制限のカウントは消費しない（処理が正常完了していないため）。

### P3A-EC-004: AI 処理中にアプリがバックグラウンドに移行した場合

- **条件**: オンデバイス LLM の処理中にユーザーがアプリをバックグラウンドに移行した場合
- **対応**: `BGTaskScheduler` を使用して処理を継続する。処理完了後、ローカル通知で「AI 分析が完了しました」と通知する。

### P3A-EC-005: メモリ不足でオンデバイス LLM が起動できない場合

- **条件**: STT モデルアンロード後もメモリに余裕がない場合（他のアプリがメモリを占有）
- **対応**: AI 処理をキューに保留し、「メモリ不足のため AI 分析を延期しました。次回アプリ起動時に再試行します」と表示する。

### P3A-EC-006: 月次リセットをまたぐ AI 処理

- **条件**: 月末に AI 処理をキューに追加し、処理完了が翌月1日以降になった場合
- **対応**: AI 処理のカウントは **キュー追加時点の月** に対して消費する（処理完了時点ではない）。これにより、月末のキューイングが翌月のカウントを圧迫しない。

### P3A-EC-007: ユーザーが文字起こしテキストを編集後に AI 再生成した場合

- **条件**: 文字起こしテキストを手動編集後、AI 要約の「再生成」をリクエストした場合
- **対応**: 編集後のテキストで AI 要約・タグを再生成する。既存の AI 要約・タグは上書きされる。月15回制限の 1 回としてカウントする（P3A-REQ-012）。
